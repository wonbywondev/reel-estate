프로젝트 개요

마지막 모델 배포하기 프로젝트에 오신 여러분들을 진심으로 축하드립니다. 이미 여러분들은 AI 엔지니어 취업에 필요한 대부분의 기술을 습득하셨어요.

혁신은 뛰어난 기술력과 번뜩이는 창의력으로부터 나온다고 합니다. 지금까지 학습하신 내용을 바탕으로, 지금까지와의 프로젝트와는 다르게 조금 더 창의적인 프로젝트를 진행해봅시다.

  • 이번 프로젝트의 목표는 생성형 AI 기술을 활용하여 소상공인이 광고 콘텐츠를 손쉽게 제작할 수 있도록 서비스를 개발하는 것이 목표입니다.
    • 디자인 역량이나 전문 도구 없이도 제품 이미지, 배너, 상세 페이지용 시각 자료, 메뉴판 이미지, 광고 문구 같은 콘텐츠를 자동으로 생성할 수 있도록 하여, 오프라인 중심의 사업자들이 온라인 마케팅에 쉽게 진입할 수 있도록 하는 것이, 서비스의 목표입니다.
  • ‘소상공인을 위한 광고컨텐츠 생성 서비스’ 라는 대주제 안에서 어떤 내용으로 세부적으로 구현할 것인지는 팀 안에서 자율적으로 결정해주세요.
    • 고정/자율 여부가 ‘고정’으로 된 항목들은 ‘가이드라인’에 적힌 내용을 반드시 따라주셔야 합니다.
    • 고정/자율 여부가 ‘자율’로 된 항목들은 ‘가이드라인’에 적힌 내용을 참고해주시거나, 완전히 자유롭게 하셔도 무방합니다.
    항목고정/자율 여부가이드라인
    서비스 이용 주체
    (유저)    		고정소상공인(직원 수가 적고, 영세한 자본으로 오프라인 또는 온라인을 통해 제품 또는 서비스를 판매하는 개인사업자 또는 법인사업자를 의미하며, 광고 및 콘텐츠 제작에 인력·예산상의 어려움을 겪는 사업자)
    서비스 제공 주체자율영리적 목적의 기업, 또는 비영리적 목적의 단체 (정부, 사회적 기업 등)
    서비스 목적고정생성형 AI 기술을 활용한 소상공인이 광고 콘텐츠를 손쉽게 제작할 수 있는 서비스
    서비스 기능자율하단의 ‘구현 기능 예시’ 참고
    (단, 최소 한 가지 기능 이상은 구현이 되어야 제출 인정. 예시 이외의 기능을 구현할 경우 예시에 준하는 기능 구현 필요)
    서비스 산출물자율제품 이미지, 배너, 상세 페이지용 시각 자료, 메뉴판 이미지, 광고 문구, 또는 이에 준하는 광고 콘텐츠
  • 프로젝트의 핵심은 ‘생성형 AI 활용’지만, 메인 결과물은 ‘서비스’를 만드는 것임을 잊지 말아주세요. 서비스 전체의 완성도를 신경 써주세요.

프로젝트 가이드

프로젝트는 아래 가이드라인에 맞추어 진행해 주세요.

심화 과제는 선택 사항으로, 필요한 경우 팀원들의 합의 하에 작업을 수행해 보세요.

요구사항 분석 및 팀 별 역할 분배
  • 프로젝트 가이드를 전반적으로 팀원들과 같이 읽어보며 어떤 작업을 수행해야하는지 파악해보세요.
  • 좀 더 체계적인 협업을 위해, 아래와 같이 역할군을 나눌 것을 권장드려요.
    • 반드시 아래 가이드라인을 따를 필요는 없어요. 팀 상황과 개인의 관심사에 맞게 역할을 자유롭게 부여하여 프로젝트를 진행해보세요.
    역할명상세
    Project Manager- 프로젝트의 협업 과정을 매니징하는 역할
    - 애자일/스프린트 방식으로 단위를 분리하고 회의를 주도
    - 서비스 기획
    - 서비스 테스트 및 평가
    모델 개발 담당- 생성 모델 선정
    - 모델 운영 파이프라인 구현
    - (필요 시)데이터 수집하여 모델 파인튜닝
    - 모델 성능 평가
    모델 서빙 담당 - 모델 서빙 전략 수립 및 구현
    - 추론 시 속도 및 메모리 최적화
    서비스 개발 담당- 서비스 요구사항에 기반해 화면 구현
    - 서비스 요구사항에 맞춰 서버 환경을 구성하고 백엔드 로직 개발
협업 타임라인 설정 및 업무 분배
  • 팀장님(또는 Project Manager)의 주도 하에 프로젝트 목표를 달성하기 위한 세부 계획을 업무 단위로 작성하고 적절하게 분배해보세요.
  • 누가 어떤 업무를 언제까지 해올지 위주로 작성하시고, 꾸준하게 트래킹하면서 프로젝트 진행상황을 체크해보세요.
협업 일지 작성
  • 팀장님(또는 Project Manager)과 모든 팀원분들은 각자 편한 방식(Notion, 구글 문서, 워드, 기타 마크다운 툴)으로 개인단위의 협업일지를 작성해주세요.
  • 매일매일 프로젝트 시작하기 전 오늘 무엇을 해야하는지, 무엇을 했고 하지 못했는지, 하루를 마무리하며 협업에 대한 소감은 어땠는지에 대해 상세하게 작성해주세요.
  • 어떤 코드를 작성했고 어떤 인사이트를 얻었고 어떤 고민을 했고 어떤 실수를 했는지까지, 최대한 구체적으로 작성할 수록 좋습니다. 무엇보다 중요한건 오늘 팀을 위해 내가 어떤 기여를 했는지 작성하는 거예요.
  • 내가 제시한 의견과 아이디어, 결과물이 팀 전체의 방향에 반영되지 않아도 괜찮아요. 일단 적어봅시다.
  • 완벽한 정리보다는 최대한 오늘의 기록을 많이 남기려고 노력해보세요. (협업 일지가 얼마나 잘 정돈되어있느냐는 평가하지 않습니다.)
  • 스프린터분들의 개인 포트폴리오 구성을 돕고자 협업일지 작성을 평가 항목으로 포함하고 있습니다. 포트폴리오 핵심인 프로젝트 파트를 스프린트가 끝난 뒤 작성하는 것은 절대 권장드리지 않습니다. 프로젝트를 할 당시의 감정과 생각을 최대한 녹여내어 매일의 일지를 작성하고, 여유가 될 때 정리합시다.
  • 협업 일지 예시는 본 문서 최하단 토글을 참조해주세요. 필수적으로 들어가야하는 항목을 확인해주세요.
프로젝트 셋업
  • 제공받은 GCP VM 내에서 Python 프로젝트 용 주요 패키지들을 설치하고, SSH 설정을 통해 협업 환경 세팅을 완료합시다.
  • Github을 이용하여 프로젝트 리포지토리를 생성하고, 팀원을 초대합니다.
  • 프로젝트 디렉토리 구조를 설정합니다.
  • 진행하면서 어려움을 겪는 팀은 주강사 및 멘토님에게 조언을 적극적으로 요청해 보세요. 주강사님은 여러분들이 프로젝트를 하시는 동안 강의실에 대기하실 거예요. 좋은 질문으로 좋은 프로젝트를 완성해갑시다.
실습 환경 및 기술 스택 안내
  • 중급 프로젝트와 동일한 스펙의 GCP 서버를 제공해드립니다.
  • 사용하실 기술 스택은 자유롭게 선택하시면 되겠습니다.
  • 이미지 생성에 HuggingFace를 활용할 수 있습니다. HuggingFace 사용법은 알고 계실 테니, 이미지 생성 관련해서 아주 간단하게 가이드를 전달드립니다.

모델 성능 및 서비스 사용성 평가

프로젝트 결과물의 평가를 위해 다음과 같은 기준으로 평가해보세요.

(강사님께서 평가하시는 평가 항목과는 달라요. 프로젝트 고도화를 위한 평가도구로 활용해보세요.)

항목질문가이드
정확도와 적합도생성된 이미지/문구가 프롬프트(스타일, 톤, 사용 목적 등)을 얼마나 잘 반영하나요?
일관성과 안정성같은 조건과 입력에 대해 생성된 콘텐츠들이 주제, 스타일, 톤에서 일관성을 유지하나요?
응답 속도와 부하 처리콘텐츠 생성에 걸리는 시간이 실제 서비스로서 만족스러운가요?
유저 중심 인터페이스유저가 자연스럽게 콘텐츠를 생성할 수 있도록 입력/결과 흐름이 설계되었나요?
서비스 완성도전체 생성 흐름이 자연스럽고, 실제 활용 가능한 수준인가요?
제어 가능성사용자가 입력이나 옵션을 조정함으로써 원하는 방향으로 결과를 유도할 수 있나요?
  • 서비스 완성도에 대한 가이드
    • Streamlit을 사용하더라도 그 한계 내에서 최대한 사용하기 편한 UI/UX를 설계해야 합니다. 사용자 입장에서 불편하거나 비직관적인 부분은 없는지 계속 고민하고 개선해보세요.
    • 다양한 사용 시나리오와 엣지 케이스를 미리 구상해 보고, 실제 사용자가 겪을 수 있는 문제들을 테스트를 통해 꾸준히 점검해야 합니다.
    • 모델 응답 속도, 이미지 데이터 전달 방식 등을 최적화해 보세요.
    • 어디서 문제가 발생하는지 관찰할 수 있도록 로깅과 모니터링을 수행합니다.
결론 도출 및 분석 보고서 작성, 시연 영상 촬영
  • 프로젝트 진행 과정, 결과, 성과 및 개선 사항, 주요 의사결정 내용들이 잘 드러나도록 정돈된 형태의 보고서를 작성해봅시다.
  • 보고서의 포맷은 자유롭게 하셔도 좋습니다. 발표자료를 보고서로 제출하셔도 괜찮아요.
  • 시연 영상은 필수가 아니에요. 제출물에 포함되지 않습니다.
    • 다만, 프로젝트 마무리 시 시연 영상을 한번 찍어두면 포트폴리오 작성 시 매우 유리해서 추천드려요.
결과 제출 및 공유(발표 세션)
  • 작성한 보고서를 토대로 프로젝트 과정 및 결과를 공유하는 팀 별 전체 발표 세션을 진행해요.
  • 발표는 팀 당 25분으로 진행합니다. (발표 20분, 질의응답 5분) 발표 시간을 초과했다고 해서 평가에 불이익은 없으나, 원활한 세션 진행을 위해 되도록이면 가이드를 지켜주세요.
  • 발표는 모든 팀원이 참여해주세요.
  • 발표 순서는 1팀부터 마지막 팀 순으로 진행합니다.
  • 상대 팀의 발표를 들으면서 궁금한 점이 있다면 Zep 강의실의 채팅 기능을 활용하여 질문해주세요. 칭찬과 격려의 코멘트도 좋습니다!
발표 후 멘토링
  • 발표세션 이후 진행되는 멘토링은 결과물에 대한 회고 멘토링으로 진행됩니다.
  • 어떤 부분이 어려웠고 준비를 잘 했는지에 대해 서로 피드백을 해보고, 다음 프로젝트에서는 어떻게 접근하면 좋을지에 대해서 의견을 나눠보세요.
  • 필요 시, 리팩토링을 진행합니다.
    • 발표 세션과 멘토링에서 파악한 피드백을 정리하고 개선사항을 정리합니다.
    • 개선사항 목록을 토대로 리팩토링 계획을 수립하고 작업을 분담합니다.
    • 리팩토링 결과 및 개선사항을 별도로 정리합니다.

이번 프로젝트 수행을 위해 GCP 서버에서 VM과 L4 GPU를 제공해드립니다.

GCP 계정- GCP를 이용하기 위한 전용 구글계정을 발급하여 부여해드릴 예정입니다.
- GCP 계정 아이디는 LMS 계정과 동일하나, 도메인이 codeit-sprint.kr로 다릅니다.
- ex) 코드잇 LMS 계정이 spai0001@codeit.com 이라면, GCP 계정은 spai0001@codeit-sprint.kr
- GCP 계정 비밀번호는 코드잇 LMS 최초 비밀번호와 동일하며, 최초 로그인 시 비밀번호를 반드시 변경하셔야 합니다.
GCP 2단계 인증- 계정 보안을 위해 2단계 인증을 반드시 설정하셔야하며, 바로 설정하지 않으시면 일정 기간 후 접속이 차단되실 수 있습니다.
- 구글 OTP를 통한 2단계 인증을 진행하시기를 권장드립니다.
GCP 프로젝트- 각 팀 별로 협업하실 수 있는 공간인 프로젝트를 만들어두었습니다.
- 구글 로그인 > GCP 접속 > 콘솔로 이동 > 프로젝트 선택 > 조직 선택 ‘codeit-sprint.kr’에서 토글을 펼치면 자동적으로 프로젝트가 보여야하는 것이 정상입니다.
- 보이지 않으면, 디스코드 공지의 팀 별 프로젝트 링크를 통해 이동해주세요.
제공 VM 사양- region
- us-central1 : 기본적으로 이 region으로 세팅하시면 됩니다.
- us-east1 : 단, sprint-ai-chunk4-0*와 같이 chunk4에 배정을 받으셨다면 해당 region으로 세팅해주셔야합니다.
- CPU : g2-standard (vCpu : 4, Memory : 16GB) x 1
- GPU : L4 x 1
- 프로젝트 종료 시점까지 해당 사양의 VM + GPU를 24시간 무제한 사용하실 수 있습니다.
- VM 생성 시 디스크 크기는 100GB 이하로 제한해주세요.
- VM 생성 시 이미지는 자유롭게 선택하되, Cuda가 설치된 이미지를 선택하시기를 권장 드려요. (ex. ubuntu-accelerator-2204-amd64-with-nvidia-550)
주의사항- 프로젝트 당 VM은 최대 한개로 유지해주세요. (추가적인 VM 생성은 별도 과금이 발생합니다.)
- 온프레미스 서버 이용 시 보안 위험을 항상 고려해주세요. 각종 API의 key, SSH key가 Github에 올라가지 않도록 보안에 주의해주세요.
- 대용량 데이터의 송.수신이 필요한 외부 Api나 도구를 직접 VM에 설치하시는 경우 네트워크 비용이 과도하게 발생할 수 있으니 본 가이드 외에 명시된 기술스택을 사용하시는 경우 반드시 주강사님 또는 운영진과의 확인을 부탁드립니다.
- 위에 명시된 내용을 포함하여 이용자의 보안 상 부주의로 비용이 과다청구될 경우 자비 부담 요청을 드릴 수 있습니다.

OpenAI Api 사용 가이드

이번 프로젝트에서도 GCP 뿐만 아니라 OpenAI Api를 제공합니다.

  • 범용적 도구로써는 gpt-5-mini, gpt-5-nano 모델 사용이 가능합니다.
  • 이미지 생성 모델은 최대한 Hugging Face 사용을 추천드리지만, 제한적으로 GPT-image-1-mini 사용도 가능합니다.
  • 텍스트 임베딩 모델은 text-embedding-3-small 모델 사용이 가능합니다.
  • Api Key는 디스코드 공지를 참고해주세요.

평가 방식

평가 방식은 크게 과정 중 평가와 결과 평가로 나뉩니다.

결과 평가의 경우 팀 단위 평가와 개인 단위 평가로 나누어 시행합니다.

  • 과정 중 평가
    • 평가자 : 각 팀의 멘토님들
    • 평가단위 : 개인 단위 평가
    • 평가항목 : 적극성, 문제 해결 능력, 의사소통 능력, 성장 가능성 (각 25점, 총 100점)
    • 평가시점 : 프로젝트 종료 후 돌아오는 일요일 23:59분까지
    • 프로젝트 기간 중 멘토링 시간 및 디스코드 채널 내 팀원 간 소통하시는 모습 등 다양한 경로를 통해 멘토님들께서 스프린터 분들의 소프트 스킬을 평가합니다.
  • 결과 평가 (팀 단위)
    • 평가자 : 주 강사님
    • 평가단위 : 팀 단위 평가
    • 평가항목 : 하단 토글 확인
      • 평가 항목 리스트
        • 프로젝트 기획 및 주제 선정
          • 프로젝트 난이도 및 요구 사항에 적합한 주제를 선정하였나요?
        • 데이터 전처리 및 탐색
          • 문제 해결에 필요한 데이터를 적절히 수집하였나요?
          • 충분한 데이터 전처리 과정을 진행하였나요?
        • 모델 선정 및 설계
          • 적절한 딥러닝 모델을 선택하고 설계하였나요?
          • 모델 설계와 선택 이유를 명확히 설명하고 있나요?
        • 모델 훈련 및 평가
          • 적절한 하이퍼파라미터 튜닝을 수행하였나요?
          • 성능을 적절한 평가 지표로 평가하였나요?
          • 성능 개선을 위해 다양한 시도를 하였나요?
        • 코드 품질 및 문서화
          • 코드가 가독성과 유지보수성을 고려하여 작성되었나요?
          • 코드 내 주석 및 마크다운이 잘 작성되어 있나요?
        • 결과 해석 및 비즈니스 시사점
          • 결과를 비즈니스 관점에서 해석하고 시사점을 도출하였나요?
          • 결과가 초기 문제 정의와 목표를 잘 충족하고 있나요?
        • 결과 발표
          • 발표에 흐름, 결과, 시사점이 잘 나타나 있나요?
          • 발표 내용을 청자들에게 효과적으로 전달하고 있나요?
    • 평가시점 : 프로젝트 최종 발표 직후
    • 프로젝트 최종 발표 내용과 보고서를 기반으로 프로젝트 결과를 종합적으로 평가합니다.
  • 결과 평가 (개인 단위)
    • 평가자 : 주 강사님
    • 평가단위 : 개인 단위 평가
    • 평가항목 : 하단 토글 확인
      • 평가 항목 리스트
        • 주제 선정
          • 프로젝트 주제 선정을 위해 유의미한 아이디어를 제시하거나, 아이디어에 대한 적절한 의견을 제시했나요?
        • 데이터 전처리 및 탐색
          • 데이터에 적합한 전처리 과정을 진행하였나요?
          • 충분한 탐색적 데이터 분석을 통해 데이터의 주요 패턴과 특징을 발견했나요?
        • 모델 훈련 및 평가
          • 모델 훈련 및 평가 과정에 있어 유의미한 기여를 했나요?
        • 업무 수행
          • 합의된 프로젝트 타임라인을 지키고 담당한 업무를 정확히 수행했나요?
        • 결과 발표
          • 발표시간 혹은 질의응답 시간에 적극적으로 참여하였나요?
    • 평가시점 : 프로젝트 최종 발표 직후
    • 개인이 작성한 협업일지, 발표 세션 참여 내역 기반으로 프로젝트 내 개인 성과를 종합적으로 평가합니다.
